How it works

A deadman's switch.
Nothing more.

No leader election. No consensus protocol. No distributed state. Just a heartbeat and a fallback.

01

Primary sends heartbeats

Your local machine sends an HTTP heartbeat to the cloud standby every 10 seconds. "I'm alive, stay idle."

02

Standby listens

The cloud instance receives heartbeats and does nothing. Zero compute. Zero cost. Just waiting.

03

Silence triggers failover

30 seconds without a heartbeat. The standby starts your service. Your agent is back online in under a minute.

04

Primary returns, standby yields

Open your laptop, heartbeats resume, the cloud gracefully stops. You're back in control. Seamless.

Quick start

Two commands. That's it.

Run one on your machine, one on your cloud instance. Done in 30 seconds.

# On your local machine (primary)

lazarus primary \
  --target https://your-standby.railway.app:8089 \
  --command "node my-agent.js"
# On your cloud instance (standby)

lazarus standby \
  --port 8089 \
  --command "node my-agent.js"
# lazarus.yml — all config in one file

role: primary

heartbeat:
  interval: 10000          # ms between heartbeats
  timeout:  30000          # ms before failover
  port:     8089
  target:   "https://standby.example.com"

service:
  command: "node server.js"

secret: "${LAZARUS_SECRET}"  # env var interpolation

Features

Small surface area.
No surprises.

Cold & warm standby

Cold: zero resources until failover. Warm: service is running, just waiting for the signal to go active.

Wrap or hook mode

Lazarus manages your process directly, or fires shell hooks for systemd/Docker/PM2 integration.

Shared secret auth

Bearer token authentication on heartbeats. Fine for private networks without it; use a secret for public endpoints.

Health endpoint

GET /health returns JSON with primary status, last heartbeat time, uptime, and role. Monitor with anything.

Config cascade

Defaults → lazarus.yml → env vars → CLI flags. Later wins. Environment variable interpolation in YAML.

Deploy anywhere

Railway, Fly.io, any VPS, Docker Compose. The standby is just an HTTP server running your command.

Honest limitations

What this is. What this isn't.

Lazarus is

  • A deadman's switch for local-first services
  • ~400 lines of TypeScript
  • One primary, one standby
  • Process coordination, not data replication
  • MIT licensed, forever

Lazarus is not

  • A distributed systems framework
  • A state replication layer
  • Multi-standby / leader election
  • A replacement for Kubernetes or Consul
  • Opinionated about your stack
